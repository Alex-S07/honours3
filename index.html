Unit 1: UI Elements & Element Identification
Understanding what a bot "sees" and how it finds things on screen
💡 Beginner Analogy

Imagine you're teaching someone to use a computer over the phone. You'd say "click the blue button that says Submit" or "type your name in the box next to the label Username." UiPath does the same — it identifies buttons, text boxes, and labels by their properties so a robot can click or type in the right place.

What are UI Elements?

UI elements are the visible, interactive parts of any software application — the things you see on screen and interact with using a mouse or keyboard.

  • Input elements — Text boxes, password fields
  • Action elements — Buttons (Submit, Login, OK)
  • Selection elements — Dropdowns, checkboxes, radio buttons
  • Display elements — Labels, tables, messages

Role in UiPath Automation

UI elements are the targets of automation. Every action your bot performs — clicking a button, typing text, reading data — happens through a UI element.

Click activities100%
Type Into activities100%
Data extraction80%

Element Identification

Element identification is the process of finding and uniquely recognizing a UI element so the bot can interact with it correctly every time, even if the screen changes slightly.

Selectors

Primary method — XML-based path to the element. Like a GPS address for UI elements.

Attributes

Properties like id, name, class that describe and uniquely identify each element.

UI Explorer

UiPath's built-in tool to inspect any element on screen and build/edit its selector.

🔍 UI Explorer — Deep Dive

UI Explorer is UiPath's dedicated inspection tool. Think of it like the browser DevTools (right-click → Inspect) but for any desktop or web application.

Main Panels

  • Visual Tree — Shows parent-child structure of the entire app
  • Properties Panel — All attributes of the selected element
  • Selector Editor — Modify the XML selector directly
  • Highlight — Draws a red box around the selected element on screen

What you can do

  • Inspect any element in any open application
  • Build and refine selectors automatically
  • Validate that a selector finds the right element
  • Edit XML directly for fine-tuning
  • Test automation compatibility before writing code

Attributes of UI Elements

Attributes are like the "identity card" of a UI element. UiPath reads these to build selectors.

AttributeMeaningReliability
idUnique identifier of the elementMost Reliable
nameLogical name assigned by developerReliable
classCSS class / type categoryModerate
titleVisible text / tooltipModerate
tagHTML tag (INPUT, BUTTON, etc.)Context-Dependent
idxIndex among duplicate elementsAvoid if possible
Example Selector XML
<!-- This tells UiPath: find an INPUT field with id='username' on a web page -->
<webctrl id='username' name='user' tag='INPUT' />
⚠️ Key Rule

Always prefer stable attributes like id and name. Avoid attributes that change dynamically (e.g., auto-generated IDs like btn_39821) — they will break your automation!

Unit 2: Selectors
The backbone of every UiPath automation workflow
💡 Beginner Analogy

A selector is like a postal address for a UI element. Just as a letter needs a country → city → street → house number to arrive at the right place, a selector contains a hierarchy: window → container → specific element. The more precise the address, the more reliably the letter (or bot action) arrives!

What is a Selector?

Selectors are XML-based strings that UiPath uses to uniquely identify and locate a UI element on screen. Every activity that interacts with an element (Click, Type Into, etc.) uses a selector internally.

Basic Selector Structure
<!-- Parent window -->
<wnd title='Login Page' cls='Chrome_WidgetWin_1'>
  <!-- Target element inside the window -->
  <webctrl id='username' tag='INPUT' />
</wnd>

Types of Selectors

Static Selectors

Use fixed, unchanging attribute values. Work perfectly for applications where the UI never changes — like most desktop apps.

  • Values do not change during execution
  • Most reliable type
  • Easiest to create (often auto-generated)
  • Best for: stable internal applications
<webctrl id='loginBtn' />

Dynamic Selectors

Used when UI element attributes change at runtime — common in modern web applications, ERPs, and CRMs.

  • Use variables to replace changing values
  • Use wildcards to match patterns
  • Essential for web apps
  • Require manual fine-tuning
<!-- Variable in selector -->
<webctrl id='"btn_" + rowId' />

🃏 Wildcards Explained

Wildcards are special symbols that let you match patterns instead of exact values — essential when part of an attribute changes each time.

Asterisk Wildcard: *

Matches zero or more characters of any kind.

<!-- Matches: user_001, user_abc, user_anything -->
<webctrl id='user_*' />

Use case: Window title changes from "File1 – Notepad" to "File2 – Notepad"

<wnd title='* – Notepad' />

Question Mark Wildcard: ?

Matches exactly one character.

<!-- Matches: btn1, btn2, btn9 (NOT btn10) -->
<webctrl id='btn?' />

Less commonly used than *, but useful for single-character variations like numbered buttons.

Full vs Partial Selectors

Full Selector

Contains the complete hierarchy from top-level window down to the target element. Used in standalone activities.

<wnd app='chrome.exe'>
  <webctrl id='submit'/>
</wnd>

Partial Selector

Contains only the target element info — used inside containers like "Attach Window" or "Open Browser" which already specify the parent.

<!-- Inside "Open Browser" container -->
<webctrl id='submit'/>

Selector Validation & Editing

Before running your automation, always validate your selectors to ensure they correctly identify the intended element.

Validation Methods

  • Click Highlight in UI Explorer — draws a red rectangle around the found element
  • Click Validate button — shows green (found) or red (not found) status
  • Run the workflow and check if the correct element is targeted

Editing Best Practices

  • Remove unnecessary attributes — simpler is more stable
  • Replace dynamic values with wildcards
  • Prefer meaningful attributes (id, name) over positional ones (idx)
  • Avoid deeply nested hierarchies — shorten where possible
  • Test in different environments (dev, test, production)

Unit 3: Advanced UI Automation
Handling dynamic elements, anchors, and fine-tuning for the real world
💡 Why "Advanced"?

Basic selectors work great for stable applications. But real-world apps — websites, ERPs, cloud platforms — change their UI constantly. This unit teaches you to build automations that adapt to those changes automatically.

Dynamic UI Elements

Dynamic UI elements are elements that change their properties between executions. This is the most common cause of broken automations.

Auto-Generated IDs

Web frameworks generate IDs like btn_3921 that change each page load.

Dynamic Tables

Table rows and cells get different indices as data changes.

Changing Positions

Elements shift position due to responsive design or added content above them.

Handling Dynamic UI — 3 Techniques

Technique 1: Wildcards in Selectors

Replace the changing part of an attribute with * or ? wildcards.

<!-- BEFORE: breaks if ID changes -->
<webctrl id='product_row_3829' />

<!-- AFTER: works for any row ID -->
<webctrl id='product_row_*' />

Technique 2: Dynamic Selectors with Variables

Store changing values in a UiPath variable and inject it into the selector at runtime.

<!-- rowNumber is a variable set at runtime -->
<webctrl id='"row_" + rowNumber.ToString()' />

Use CTRL+K inside the Selector Editor to create a variable directly in the selector field.

Technique 3: Anchor-Based Automation ⚓

When a target element has no stable attributes, use a nearby stable element (the anchor) to locate it by proximity.

Label: "Username"
(stable anchor)
→ points to →
Input Field
(unstable target)

The bot finds the label "Username" (which never changes), then looks for the input field next to it. Even if the input field's ID changes, the anchor stays the same.

✅ Real-World Example

A billing form has two identical text boxes for "billing address" and "shipping address." Their IDs are the same! Use the visible labels as anchors to tell them apart.

Selector Fine-Tuning

Fine-tuning is the process of improving a selector to make it work across all environments reliably, without being so rigid that minor UI changes break it.

Reliability Checklist

  • ✅ Uses stable attributes (id, name, automationId)
  • ✅ No unnecessary attributes in the chain
  • ✅ Wildcards replace dynamic values
  • ✅ Validates in development AND production
  • ✅ Short, simple hierarchy
  • ❌ No idx unless value is very small (1 or 2)
  • ❌ No fully-wildcard attributes (name="*" — remove instead)

Why Reliability Matters

A fragile selector means:

  • Bot fails silently in production
  • Hours of debugging time wasted
  • Business processes halt unexpectedly
  • Loss of trust in automation

A reliable selector finds the same element every time — in dev, test, and production, regardless of who is logged in.

Unit 4: Screen Scraping & OCR
Extracting text and data from screens, images, and scanned documents
💡 What is Screen Scraping?

Screen scraping is like taking a photo of a screen and then reading all the text from it. It's used when you can't simply "get" a value from a database — you need to read it directly from what's displayed in the application.

Screen Scraping Methods

UiPath provides 3 methods to extract text from UI. Each works differently and suits different situations:

⚡ Full Text

Fastest Background
  • Hooks directly into the rendering engine
  • Extracts all text at once
  • Works in background (no visible window needed)
  • Does NOT preserve layout/formatting
  • Not supported in all apps

Best when: you only need raw text, speed matters

🖥️ Native

Medium Speed Structure
  • Works with most desktop applications
  • Preserves text position and layout
  • Usually needs app in foreground
  • Good accuracy
  • Better than Full Text for positional data

Best when: position/format of text matters

📷 OCR

Slower Image-Based
  • Works on screenshots, images, scanned docs
  • Works in Citrix / virtual environments
  • Accuracy depends on image quality
  • Not 100% accurate — always verify critical output
  • Last resort when other methods fail

Best when: text is in an image, Citrix, PDF scan

Full Text vs Native — Comparison Table

Feature Full Text Scraping Native Scraping
Speed ⚡ Very fast 🐢 Medium
Accuracy Very high (for text) High
Background Execution ✔ Works in background ✘ Usually needs foreground
Text Extraction Type Entire text block at once Text with structure / position
Preserves Formatting ✘ Does NOT preserve layout ✔ Preserves layout
Supported Applications Limited (not all apps) Works with most desktop apps
Best Use Case When only plain text needed When position/format matters

OCR Engines in UiPath

UiPath supports multiple OCR engines. Each has strengths for different scenarios:

🔬 Google OCR (Tesseract)

Free Local Processing
  • Open-source engine (Tesseract v4.0)
  • Uses LSTM neural network for recognition
  • Processing done locally on your machine
  • Good for small text areas
  • Supports multiple languages
  • Free to use — comes with UiPath Studio

💻 Microsoft OCR (MODI)

Free Large Areas
  • Microsoft Document Imaging OCR
  • Processing done locally
  • Better for larger text areas
  • Warning: can cause memory leaks in loops
  • For loop use: run in separate invoked workflow with Isolated=true
  • Free — included with UiPath

☁️ Google Cloud OCR

Paid Cloud
  • Uses Google Cloud Vision API
  • Image uploaded to Google servers — not local
  • Faster and more precise than Tesseract
  • Requires API key (free trial available)
  • Best for high-accuracy cloud workflows

🏆 ABBYY OCR

Licensed Most Accurate
  • Industry-leading accuracy
  • Supports 200+ languages including Asian scripts
  • Handles printed AND handwritten text
  • Zonal OCR for specific field extraction
  • Great for invoices, forms, receipts
  • Requires paid license from ABBYY / UiPath

Key OCR Properties (Tesseract)

PropertyWhat it does
ScaleEnlarges the image before OCR. Higher = better accuracy for small text. Default: 2
ProfileNone = no preprocessing; Screen = for remote desktop; Scan = for scanned files
InvertInverts colors — use when text is light on dark background
LanguageLanguage to use for recognition. Write "English" (full name)
ExtractWordsIf checked: also returns the screen coordinates of each word found
AllowedCharsOnly extract specific characters (e.g., digits only: "0123456789")

Digitize paper archives, extract tabular data, populate databases without manual entry.

⚠️ OCR Limitations — Important for Exams

OCR is NOT 100% accurate. Accuracy depends on: image quality, font clarity, lighting, resolution, and whether the document is skewed. Always validate critical extracted data. Low-quality scans produce errors. Use image pre-processing (Scale, Invert, Profile) to improve results.

Quick Summary & Exam Tips
Everything you need to remember for your exam

Unit 1 — Key Points

  • UI elements = visible interactive parts of any app
  • Element identification = locating elements precisely
  • UI Explorer = inspection tool for building selectors
  • Best attributes: id (most reliable), name, class
  • Avoid: auto-generated IDs, idx attribute

Unit 2 — Key Points

  • Selectors = XML fragments that identify UI elements
  • Static selector = fixed values, most reliable
  • Dynamic selector = uses variables or wildcards
  • * wildcard = matches zero or more characters
  • ? wildcard = matches exactly one character
  • Full selector = includes top-level window
  • Partial selector = used inside containers

Unit 3 — Key Points

  • Dynamic elements change attributes at runtime
  • Fix with: wildcards, dynamic variables, anchors
  • Anchor = stable nearby element to locate a target
  • Fine-tune: remove excess attributes, shorten hierarchy
  • Never use idx > 2, never use full-wildcard attributes

Unit 4 — Key Points

  • Screen scraping = extracting visible text from screens
  • Full Text = fastest, background, no layout
  • Native = medium, preserves structure
  • OCR = images/scans/Citrix, slowest, not 100% accurate
  • Tesseract = Google/free/local | MODI = Microsoft/local
  • ABBYY = most accurate, paid, 200+ languages
  • OCR steps: Capture → Detect → Recognize → Convert
🎓 Module Project Reminder

The module project asks you to build an automation process that extracts screen or document data and enters it into an application using reliable UI selectors. This means you'll need to combine: UI element identification + well-tuned selectors + screen scraping or OCR + data entry via Type Into activities.
